# Agent 工具设计

## 概述

LLM Agent 通过调用工具与数据库环境交互，完成参数调优。工具分为三类：**观察**（了解环境）、**行动**（修改配置）、**验证**（评估效果）。

## 工具列表

### 观察类

#### `get_hardware_info`
获取硬件环境信息。

```
输入: 无
输出: {cpu_count, cpu_model, total_memory_gb, disk_type, ...}
```

用途：Agent 根据硬件决定参数上限（如 shared_buffers 不能超过物理内存）。

---

#### `get_current_config`
获取当前 knob 配置。

```
输入: knob_names (可选，默认返回所有可调 knob)
输出: {knob_name: current_value, ...}
```

用途：了解当前配置，作为调优起点。

---

#### `get_db_metrics`
获取 DB 运行时指标。

```
输入: 无
输出: {
  buffer_hit_rate: 0.85,
  cache_hit_rate: 0.92,
  dead_tuple_ratio: 0.15,
  temp_files_count: 234,
  seq_scan_ratio: 0.6,
  ...
}
```

用途：Agent 通过指标诊断瓶颈。例如 buffer_hit_rate 低 → shared_buffers 不够；temp_files 多 → work_mem 太小。

---

#### `get_workload_info`
获取 workload 特征。

```
输入: 无
输出: {
  workload_type: "OLTP",
  read_write_ratio: "7:3",
  data_size_gb: 10,
  active_connections: 50,
  top_queries: [...],
}
```

用途：不同 workload 需要不同调优策略（OLTP 重延迟，OLAP 重吞吐）。

---

#### `get_recent_logs`
获取最近 PG 日志。

```
输入: lines (返回行数，默认 50), level (最低日志级别: WARNING/ERROR)
输出: 日志文本
```

用途：发现慢查询（duration > xxx ms）、临时文件（temporary file）、死锁（deadlock detected）、OOM 等问题，辅助诊断。

---

### 行动类

#### `set_knob`
设置数据库参数。

```
输入: knobs ({knob_name: value, ...})
输出: {
  success: [已生效的 knob],
  pending_restart: [需要重启才能生效的 knob],
  failed: [设置失败的 knob 及原因],
}
```

内部调用 `ALTER SYSTEM SET` + `pg_reload_conf()`，并返回哪些需要重启。

---

#### `restart_pg`
重启 PostgreSQL 服务。

```
输入: 无
输出: {success: bool, duration_seconds: float}
```

用途：使 static knob（如 shared_buffers、max_connections）生效。

---

#### `reload_pg`
重载 PG 配置（不重启）。

```
输入: 无
输出: {success: bool}
```

用途：dynamic knob 修改后立即生效，无需重启。

---

#### `reset_config`
恢复所有参数到默认值。

```
输入: 无
输出: {success: bool}
```

用途：回滚失败配置、实验对照。

---

### 验证类

#### `predict_performance`
用 Cost Model 预测性能（训练时用）。

```
输入: 无 (使用当前配置 + 环境特征)
输出: {
  predicted_tps: 2500.0,
  predicted_latency_avg: 3.2,
  confidence: 0.85,
}
```

用途：快速评估配置效果，无需真实跑 benchmark。训练阶段的核心工具。

---

#### `run_benchmark`
运行真实 benchmark（验证时用）。

```
输入: duration (秒，默认 60)
输出: {
  tps: 2450.3,
  latency_avg: 3.5,
  latency_p95: 8.2,
  duration: 60,
}
```

用途：真实环境验证。训练时不用（太慢），验证时使用。

---

## Agent 交互示例

```
System: 请优化以下 PostgreSQL 数据库的参数配置以提升性能。

Agent: [think] 先了解硬件和当前状态。

Agent: [tool_call] get_hardware_info()
Observation: {cpu_count: 8, total_memory_gb: 32, disk_type: "SSD"}

Agent: [tool_call] get_current_config()
Observation: {shared_buffers: "128MB", work_mem: "4MB", random_page_cost: "4.0", ...}

Agent: [tool_call] get_db_metrics()
Observation: {buffer_hit_rate: 0.72, temp_files_count: 1580, seq_scan_ratio: 0.45}

Agent: [tool_call] get_recent_logs()
Observation: WARNING: temporary file size 125MB
            LOG: duration: 2500 ms  statement: SELECT ...

Agent: [think] 分析发现：
  1. buffer_hit_rate=0.72 偏低，shared_buffers=128MB 远小于 32GB 内存
  2. temp_files 大量出现，work_mem=4MB 太小
  3. 磁盘是 SSD，random_page_cost=4.0 太高

Agent: [tool_call] set_knob({
  "shared_buffers": "8GB",
  "work_mem": "128MB",
  "effective_cache_size": "24GB",
  "random_page_cost": "1.1"
})
Observation: {success: ["work_mem", "effective_cache_size", "random_page_cost"],
              pending_restart: ["shared_buffers"]}

Agent: [tool_call] restart_pg()
Observation: {success: true, duration_seconds: 3.2}

Agent: [tool_call] predict_performance()
Observation: {predicted_tps: 2850, predicted_latency_avg: 2.8}

Agent: [answer] 推荐配置及理由：
  - shared_buffers: 128MB → 8GB（物理内存 25%，提升缓冲命中率）
  - work_mem: 4MB → 128MB（消除临时文件）
  - effective_cache_size: 4GB → 24GB（物理内存 75%）
  - random_page_cost: 4.0 → 1.1（SSD 随机读很快）
  预测 TPS 从 1200 提升到 2850（+137%）
```

## 训练 vs 验证模式

| | 训练（GRPO） | 验证 |
|---|---|---|
| 观察类工具 | 模拟环境返回 | 真实 PG 返回 |
| `set_knob` | 记录配置 | ALTER SYSTEM SET |
| `restart_pg` | 模拟（instant） | 真实重启 |
| `predict_performance` | ✅ Cost Model 预测 | ❌ 不用 |
| `run_benchmark` | ❌ 不用 | ✅ 真实 benchmark |
