# db-opt-r1 需求分析

## 一、项目背景

数据库参数（knob）调优是 DBA 的核心工作之一。一个 PostgreSQL 实例有 **270+ 个可配置参数**（如 `shared_buffers`、`work_mem`、`effective_cache_size` 等），不同的参数组合对性能（throughput、latency）影响巨大。

现有方案的问题：
- **人工调优**：依赖 DBA 经验，耗时且难以覆盖参数间的复杂交互
- **传统 RL 方案**（CDBTune、OtterTune）：黑盒模型，无可解释性；每次验证需重启 DB + 跑 benchmark，训练极慢
- **LLM Agent 方案**（AgentTune）：有推理能力，但依赖 prompt engineering，未经 RL 训练

## 二、项目目标

用 GRPO 训练一个 LLM Agent，使其能通过工具调用自主调优数据库参数。训练阶段用 Cost Model 加速，验证阶段用真实数据库确认效果。

## 三、核心需求

### 3.1 Cost Model

| 需求 | 说明 |
|------|------|
| 训练数据采集 | 在真实 PG 上采集 (knob 配置, DB metrics, 性能指标) 三元组 |
| 采集特征 | 参考 OtterTune：`SHOW ALL` 采 knob + 8 个 `pg_stat` 视图采 200+ metrics |
| 模型选择 | 优先 XGBoost（稳定、快速、对表格数据效果好） |
| 预测目标 | 输入 knob 配置 + workload 特征 → 输出预测 throughput / latency |
| 推理速度 | < 10ms（替代真实 benchmark 的分钟级执行） |

### 3.2 LLM Agent

| 需求 | 说明 |
|------|------|
| 基座模型 | Qwen2.5-Coder 或同等规模模型 |
| 交互方式 | 多轮工具调用（think → tool_call → observe → think → answer） |
| 工具集 | 待定（至少包括：查看当前配置、应用配置、查看性能指标） |
| 输出 | 推荐的 knob 配置 + 调优理由 |

### 3.3 训练

| 需求 | 说明 |
|------|------|
| 训练算法 | GRPO（复用 verl 框架） |
| Reward 来源 | 训练时用 Cost Model 预测值；验证时用真实 benchmark 结果 |
| 数据 | 需要构造 (prompt, workload 描述, 当前配置) 作为训练样本 |
| 硬件 | 2×4090（24G），需考虑显存限制 |

### 3.4 验证

| 需求 | 说明 |
|------|------|
| 数据库 | PostgreSQL（优先支持） |
| Benchmark | TPC-H / YCSB / sysbench |
| 评估指标 | throughput (tps)、latency (p95)、与默认配置的提升比 |
| 对比基线 | PG 默认配置、OtterTune、CDBTune、AgentTune |

## 四、关键挑战

1. **Cost Model 精度 vs 泛化性**：单一环境训练的 cost model 能否在不同 workload/硬件上有效？
2. **参数空间巨大**：270+ knob 中哪些值得调？需要 knob 筛选机制
3. **验证耗时**：每次真实验证需 1-3 分钟（重启 DB + 跑 benchmark），数据采集慢
4. **Reward 信号稀疏**：一组配置只有一个 throughput 值，不像编译器任务有中间 reward

## 五、参考项目

| 项目 | 参考点 |
|------|--------|
| Agent-R1 | GRPO 训练框架、多轮工具调用、Reward 计算 |
| OtterTune | 数据采集脚本、特征工程（Factor Analysis） |
| CDBTune | RL 调优流程、environment 设计 |
| DBTune | 多种代理模型、Workload 支持 |
| AgentTune | LLM Agent 调优范式 |
