# Agent 终止协议设计

## 概述

当前 `db-opt-r1` 的 agent rollout 在“何时结束”这件事上存在协议不清的问题：

- eval 侧把多轮 messages 压平成单条 user 文本，训练/评估分布不一致
- 模型没有显式结束动作，只能靠“继续调工具”或“输出坏格式”来停
- 没有 `<tool_call>` 会被当成格式错误，而不是自然结束
- `predict_performance` 没有预算约束，模型可能反复试探
- 结束原因没有结构化记录，debug 时只能靠人工猜

本设计的目标是把 agent 终止逻辑收敛成一套可训练、可评估、可调试的统一协议。

## 设计目标

终止协议满足以下要求：

1. 模型有明确的、可学习的主结束路径
2. 模型偶发偏离协议时，环境可以自然收口，不把“想停”误判成错误
3. 任务特定预算约束由环境硬控制，而不是只依赖 prompt
4. 非理想结束不会污染轨迹，也不会伪造不存在的对话
5. 训练、eval、debug 使用同一套终止语义

## 协议总览

终止逻辑按优先级分为三层：

### 第一层：显式结束

模型显式调用结束动作：

```xml
<tool_call>{"name":"finish_tuning","arguments":{}}</tool_call>
```

这是主结束路径。模型应该通过这个动作表达：

- 当前配置已经足够好
- 不需要继续调用工具
- 本轮调优到此结束

环境收到后直接：

- 不再执行额外 DB 操作
- 标记 `done=True`
- 记录 `termination_reason=finish_tuning`

### 第二层：自然结束

如果 assistant 本轮输出里没有 `<tool_call>`，则视为自然结束，而不是格式错误。

这是一条容错路径，不是鼓励模型沉默结束。它的作用是：

- 避免“模型想停”被当成协议错误
- 避免向上下文中注入大量 `Invalid tool call format`
- 避免错误重试链把后续轨迹污染掉

环境行为：

- 保留本轮 assistant 原始输出
- 不补伪造的结束语
- 直接结束 episode
- 记录 `termination_reason=no_tool_call`

### 第三层：边界结束

用于兜底和任务预算控制：

- `predict_performance` 最多 3 次
- 超过 `max_turns` 时结束

其中：

- `predict_performance` 用满 3 次属于任务规则完成，记为 `predict_budget_exhausted`
- 到达 `max_turns` 属于非理想结束，记为 `max_turns_reached`

## finish_tuning 设计

`finish_tuning` 是一个无参数、无副作用的显式结束工具：

```json
{"name":"finish_tuning","arguments":{}}
```

设计原则：

- **不携带最终 knobs 参数**
  原因是当前环境状态本身就是最终状态，避免一份状态在 env、一份状态在 `arguments` 中双写。
- **不触发新的评估或配置变更**
  `finish_tuning` 只负责“结束”，不负责“修改”。
- **与已有工具职责解耦**
  - `set_knob` 负责调整配置
  - `predict_performance` 负责评估当前配置
  - `finish_tuning` 负责声明本轮结束

一条理想轨迹示例：

```text
assistant: <think>先查看当前配置</think>
assistant: <tool_call>{"name":"get_current_config","arguments":{}}</tool_call>
user: <tool_response>...</tool_response>

assistant: <think>将 shared_buffers 调整为 3GB</think>
assistant: <tool_call>{"name":"set_knob","arguments":{"name":"shared_buffers","value":"3GB"}}</tool_call>
user: <tool_response>...</tool_response>

assistant: <think>预测当前配置性能</think>
assistant: <tool_call>{"name":"predict_performance","arguments":{}}</tool_call>
user: <tool_response>{"improvement_pct": 6.1, ...}</tool_response>

assistant: <think>当前结果已经优于基线，结束本轮调优</think>
assistant: <tool_call>{"name":"finish_tuning","arguments":{}}</tool_call>
```

## 预算规则：predict_performance 最多 3 次

该任务的目标是“通过有限尝试预测性能并确定较优配置”，不是无限制试探。

因此规定：

- 每个 episode 最多允许 3 次 `predict_performance`
- 第 3 次 `predict_performance` 的工具响应返回后，episode 正常结束
- 不再等待模型额外输出结束动作

这不是“截断”，而是任务预算用尽后的正常完成。

结束后记录：

- `termination_reason=predict_budget_exhausted`
- `predict_calls_used=3`

对应轨迹应停在第 3 次 `predict_performance` 的 `<tool_response>` 之后，不补 assistant 收尾。

## 异常结束设计

### 结束原因分类

终止原因分为三类：

#### 正常结束

- `finish_tuning`
- `no_tool_call`
- `predict_budget_exhausted`

#### 非理想边界结束

- `max_turns_reached`
- `length_exceeded`
- `repeated_tool_call`

#### 协议/执行错误

- `invalid_tool_call`
  - `<tool_call>` 结构不合法
  - 不是合法 JSON
  - 缺少 `name`
  - 工具名不存在
- `tool_execution_error`
  - 工具名合法，但参数错误
  - 工具执行过程抛错

### 处理原则

所有非正常收口都遵循同一原则：

- 保留原始轨迹，不删改历史
- 不补假的 `finish_tuning`
- 不补模型未实际输出的“总结语”
- 用**最后有效配置状态**做结算

例如：

1. `set_knob(shared_buffers=3GB)` 成功
2. `predict_performance` 成功
3. 模型输出坏掉的 `<tool_call>`

则：

- episode 停在第 3 步
- `termination_reason=invalid_tool_call`
- 最终 reward 按第 2 步之后的当前 env 配置结算

## 坏格式重试策略

对于坏格式，不采用“一次出错立即结束”的策略，而采用“提示修正 + 有上限重试”的方式。

### 坏格式提示

当前提示保持为：

```text
Invalid tool call format. Please use <tool_call>{"name": "tool_name", "arguments": {...}}</tool_call>
```

### 推荐策略

- 第一次坏格式：返回错误提示，允许下一轮重试
- 连续坏格式达到阈值后结束 episode

推荐阈值：

- `max_invalid_tool_call_streak = 2`

即：

- 第 1 次坏格式：给修正机会
- 第 2 次连续坏格式：结束，`termination_reason=invalid_tool_call`

这样做的原因是：

- 与 `Search-R1` 的“错误提示后继续试”思路对齐
- 避免无限把错误提示灌回上下文
- 让模型有一次明确的自我修正机会

## 重复工具调用限制

除了 `predict_performance` 的总预算限制外，还需要防止模型陷入“重复调用同一个工具”的局部循环。

### 规则定义

不采用“同名工具连续调用三次就结束”的粗粒度规则，而采用更精确的调用指纹规则：

- 若 **相同工具名 + 相同参数** 连续调用达到阈值，则结束 episode

推荐阈值：

- `max_same_tool_same_args_streak = 3`

结束原因记录为：

- `termination_reason=repeated_tool_call`

### 设计理由

仅按工具名限制会误伤合法策略。例如：

- `set_knob(shared_buffers=4GB)`
- `set_knob(effective_cache_size=8GB)`
- `set_knob(work_mem=32MB)`

这虽然是连续三次 `set_knob`，但不是循环，而是正常多参数调优。

真正应判定为循环的是：

```json
{"name":"get_hardware_info","arguments":{}}
{"name":"get_hardware_info","arguments":{}}
{"name":"get_hardware_info","arguments":{}}
```

或：

```json
{"name":"set_knob","arguments":{"name":"shared_buffers","value":"8GB"}}
{"name":"set_knob","arguments":{"name":"shared_buffers","value":"8GB"}}
{"name":"set_knob","arguments":{"name":"shared_buffers","value":"8GB"}}
```

### 与 predict_performance 的关系

`predict_performance` 不使用上述“相同参数重复调用”规则，而是单独按总预算控制：

- 每个 episode 最多 3 次
- 第 3 次返回结果后正常结束
- `termination_reason=predict_budget_exhausted`

## termination_reason 设计

每条 rollout 结束时记录一个明确的结束原因，用于 debug 和统计分析。

建议枚举值：

- `finish_tuning`
- `no_tool_call`
- `predict_budget_exhausted`
- `max_turns_reached`
- `length_exceeded`
- `repeated_tool_call`
- `invalid_tool_call`
- `tool_execution_error`

该字段只属于环境/rollout 元信息，不属于模型输入协议。

用途：

1. Debug：快速看出一条轨迹为什么结束
2. 训练分析：统计正常结束与非理想结束的占比
3. Eval 诊断：判断当前策略是否主要通过 `finish_tuning` 正常收口，还是频繁撞上边界/错误

## Eval 输入协议修正

当前 eval 侧将多轮 `messages` 压平成一条 user 文本，导致：

- system / user / assistant / tool 的角色边界丢失
- 训练分布与评估分布不一致
- 后续轮次更容易出现格式漂移、空 `<think>`、错误 tool call

修正方案：

- `MultiProviderLLMClient.generate()` 支持直接接收 `list[dict]` messages
- eval sampler 在拿到多轮 messages 时，原样传给 chat API
- 保留字符串 prompt 路径，用于非对话式调用场景

这项修改不直接定义“怎么结束”，但会显著影响结束协议的稳定性，因此属于同一设计的一部分。

## 数据生成逻辑修正

当前数据生成中存在一条会把模型教坏的逻辑：

- 如果轨迹以 `tool` 角色结尾，会自动补一条“调优流程已完成”的 assistant

这会把“无工具调用的结束语”混进训练分布，而 rollout 协议又要求 assistant 继续输出工具调用，形成冲突。

修正方案：

- 删除自动补的结束 assistant
- 后续结束统一由：
  - `finish_tuning`
  - 自然结束
  - 预算结束
  - 边界结束

来表达

## 轨迹与日志表现

终止后的轨迹保留原始对话顺序，不伪造对话。

建议在 debug rollout 额外记录以下元信息：

- `termination_reason`
- `predict_calls_used`
- `turn_count`
- `last_valid_tool_call`
- `last_valid_config`

其中：

- `termination_reason` 是最重要的主诊断字段
- `last_valid_config` 用于对照 reward 结算状态

日志中应当区分：

- 对话内容
- 元信息

不要把元信息伪装成 assistant / user 对话的一部分。

## 与现有项目的对齐关系

本设计不是照抄单个项目，而是对现有先例的工程化综合。

### Search-R1

- 显式终止动作：`<answer>...</answer>`
- `max_turns` 兜底
- 非法动作会给错误提示并继续尝试

### Agent-R1

- 由 env 自定义 stop 规则
- 常见实现是“没有 tool call / code block 就自然结束”
- 还受 `max_turns`、长度上限约束

### Tool-Star

- 工具闭标签触发执行（如 `</search>`、`</python>`）
- 工具调用预算耗尽后强制 EOS 收口
- EOS / length 也可结束

### DeepSearch

- 不是线性 tool agent
- 通过 `terminate_num`、`max_depth`、`is_terminal()` 控制搜索树结束

### 本项目的组合方式

`db-opt-r1` 适合组合成：

- 显式 `finish_tuning` 作为主路径
- 无 `tool_call` 作为容错路径
- `predict_performance` 3 次预算作为任务规则
- 重复调用同一工具同一参数时结束局部循环
- `max_turns` / 长度上限作为边界兜底

## 实施范围

本设计对应 6 个改动点：

1. eval 改成真正的 chat messages 输入
2. 新增显式结束动作 `finish_tuning`
3. 无 `tool_call` 改为自然结束
4. `predict_performance` 增加 3 次预算规则
5. 异常结束统一收口，但不伪造轨迹
6. 删除会把模型教坏的数据生成逻辑

## 验证目标

设计落地后，应满足以下可验证结果：

1. 正常样本能通过 `finish_tuning` 结束
2. 没有 `<tool_call>` 时不再出现 `Invalid tool call format` 重试链
3. 第 3 次 `predict_performance` 后 episode 正常结束
4. eval 不再把多轮 messages flatten 成单字符串
5. debug rollout 能明确显示每条轨迹的 `termination_reason`
6. SFT / eval / rollout 的终止语义保持一致
