# Cost Model 特征设计

## 概述

Cost Model 的输入特征分为四大类：**硬件特征**、**软件特征**、**Knob 配置**、**DB 运行时指标**。

前两类描述"环境是什么样的"，后两类描述"在这个环境下做了什么、发生了什么"。

---

## 一、硬件特征

描述运行数据库的物理/虚拟硬件环境。相同的 knob 配置在不同硬件上效果差异巨大。

| 特征 | 采集方式 | 示例值 | 影响 |
|------|---------|--------|------|
| CPU 核数 | `nproc` / `os.cpu_count()` | 8 | 并行查询、vacuum worker 数量 |
| CPU 型号 | `/proc/cpuinfo` | Intel i7-12700 | 单核性能差异 |
| CPU 频率 (MHz) | `/proc/cpuinfo` | 3600 | 计算密集型查询性能 |
| 总内存 (GB) | `free -g` / `psutil.virtual_memory()` | 32 | shared_buffers、work_mem 上限 |
| 可用内存 (GB) | 同上 | 24 | 实际可分配给 DB 的内存 |
| 磁盘类型 | `lsblk -d -o name,rota` | SSD(0) / HDD(1) | random_page_cost 取值 |
| 磁盘读带宽 (MB/s) | `fio` / 历史 benchmark | 500 | IO 密集型查询瓶颈 |
| 磁盘写带宽 (MB/s) | 同上 | 400 | WAL 写入、checkpoint |
| 磁盘容量 (GB) | `df -h` | 500 | temp_file 空间 |

## 二、软件特征

描述数据库软件和操作系统环境。

| 特征 | 采集方式 | 示例值 | 影响 |
|------|---------|--------|------|
| PostgreSQL 版本 | `SELECT version()` | 16.2 | 不同版本优化器行为不同 |
| OS 类型 | `uname -s` | Linux | 系统调用差异 |
| OS 版本 | `cat /etc/os-release` | Ubuntu 22.04 | 内核调度策略 |
| 内核版本 | `uname -r` | 5.15.0-91 | IO 调度、大页支持 |
| 文件系统 | `df -T` | ext4 / xfs | IO 性能特征 |
| huge_pages 状态 | `cat /proc/meminfo` | on / off | 大内存分配效率 |
| 数据库编码 | `SHOW server_encoding` | UTF8 | 字符处理开销 |
| 最大连接数设置 | `SHOW max_connections` | 100 | 并发能力上限 |

## 三、Knob 配置（输入变量）

这是 cost model 主要的输入维度——Agent 要调的参数。

### 3.1 关键 Knob（优先调优）

参考 OtterTune 和 CDBTune 的研究，以下 knob 对性能影响最大：

| Knob | 类型 | 默认值 | 作用 |
|------|------|--------|------|
| `shared_buffers` | 内存 | 128MB | 共享缓冲区大小 |
| `work_mem` | 内存 | 4MB | 排序/哈希操作内存 |
| `effective_cache_size` | 内存 | 4GB | 优化器对系统缓存的估计 |
| `maintenance_work_mem` | 内存 | 64MB | 维护操作（VACUUM）内存 |
| `wal_buffers` | 内存 | -1(auto) | WAL 缓冲区 |
| `random_page_cost` | 优化器 | 4.0 | 随机IO代价估计 |
| `seq_page_cost` | 优化器 | 1.0 | 顺序IO代价估计 |
| `effective_io_concurrency` | IO | 0 | 并发 IO 操作数 |
| `max_parallel_workers_per_gather` | 并行 | 2 | 每次查询最大并行 worker |
| `max_parallel_workers` | 并行 | 8 | 全局最大并行 worker |
| `checkpoint_completion_target` | WAL | 0.5 | checkpoint 平滑度 |
| `max_wal_size` | WAL | 1GB | WAL 最大尺寸 |
| `bgwriter_delay` | 后台写 | 200ms | 后台写入间隔 |
| `bgwriter_lru_maxpages` | 后台写 | 100 | 每轮最大写入页数 |

### 3.2 全量采集

通过 `SHOW ALL` 可获取 270+ 个参数，全量作为特征输入，交给模型自动学习重要性。

## 四、DB 运行时指标（Workload 特征）

描述"在某组 knob 下跑了 workload 之后，DB 内部发生了什么"。参考 OtterTune 的 `PostgresCollector`。

### 4.1 全局指标

| 来源视图 | 关键指标 | 含义 |
|---------|---------|------|
| `pg_stat_bgwriter` | buffers_checkpoint, buffers_clean, buffers_alloc, buffers_backend | 缓冲区活动：checkpoint 写出、后台清理、分配、backend 直接写 |
| `pg_stat_archiver` | archived_count, failed_count | WAL 归档情况 |

### 4.2 数据库级指标

| 来源视图 | 关键指标 | 含义 |
|---------|---------|------|
| `pg_stat_database` | xact_commit, xact_rollback | 事务提交/回滚数 |
| | tup_returned, tup_fetched | 返回/获取的行数 |
| | tup_inserted, tup_updated, tup_deleted | 增删改行数 |
| | blks_hit, blks_read | 缓冲区命中 / 磁盘读取 |
| | temp_files, temp_bytes | 临时文件使用 |
| | deadlocks | 死锁次数 |
| `pg_stat_database_conflicts` | confl_lock, confl_snapshot, confl_deadlock | 各类冲突计数 |

### 4.3 表级指标

| 来源视图 | 关键指标 | 含义 |
|---------|---------|------|
| `pg_stat_user_tables` | seq_scan, idx_scan | 全表扫描 / 索引扫描次数 |
| | n_tup_ins, n_tup_upd, n_tup_del | 行级增删改 |
| | n_live_tup, n_dead_tup | 活/死元组数 |
| `pg_statio_user_tables` | heap_blks_hit, heap_blks_read | 堆页缓冲命中 / 磁盘读取 |
| | idx_blks_hit, idx_blks_read | 索引页缓冲命中 / 磁盘读取 |

### 4.4 索引级指标

| 来源视图 | 关键指标 | 含义 |
|---------|---------|------|
| `pg_stat_user_indexes` | idx_scan, idx_tup_fetch, idx_tup_read | 索引使用情况 |
| `pg_statio_user_indexes` | idx_blks_hit, idx_blks_read | 索引 IO 情况 |

## 五、预测目标（Label）

| 指标 | 含义 | 来源 |
|------|------|------|
| throughput (tps) | 每秒事务数 | Benchmark 工具输出 |
| latency_avg (ms) | 平均延迟 | Benchmark 工具输出 |
| latency_p95 (ms) | P95 延迟 | Benchmark 工具输出 |
| latency_p99 (ms) | P99 延迟 | Benchmark 工具输出 |

## 六、特征处理策略

| 策略 | 应用范围 |
|------|---------|
| **归一化** | 所有数值型 knob 和 metrics |
| **One-hot** | 磁盘类型(SSD/HDD)、OS 类型、文件系统类型 |
| **对数变换** | 内存类参数（shared_buffers 等，跨越 MB→GB 量级） |
| **差值计算** | 运行时指标取 benchmark 前后的差值（参考 OtterTune） |
| **Factor Analysis** | 200+ 运行时指标降维到 ~10 维（参考 OtterTune） |
