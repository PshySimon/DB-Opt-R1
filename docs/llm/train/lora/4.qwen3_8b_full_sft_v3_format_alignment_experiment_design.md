# Qwen3-8B Full SFT v3 格式对齐实验设计

## 背景

前两轮 Qwen3-8B SFT 结果已经形成几个稳定事实：

| 模型 | DB avg vs 原始配置 | repetition | trunc/protocol | predict_rate | set_knob_rate | finish_rate | BFCL core |
|------|-------------------:|-----------:|---------------:|-------------:|--------------:|------------:|----------:|
| `qwen3_8b_base` | 1.84% | 59.33% | 99.83% | 88.33% | 78.67% | 29.67% | 41.56% |
| `lora_d4_6k_ckpt537` | 8.81% | 0.67% | 0.67% | 99.33% | 98.00% | 92.00% | 43.58% |
| `full_8b_ckpt350` | 5.45% | - | - | - | - | - | 21.28% |
| `full_6p6k_lr1e6_ep1` | - | 83.50% | 70.17% | 12.17% | 4.67% | 0.17% | - |
| `full_d2_3k_lr1e5_ep3_ckpt150` | 3.92% | 4.67% | 6.83% | 77.33% | 98.50% | 58.00% | - |
| `full_d2_3k_lr1e5_ep3_ckpt250` | 4.07% | 9.83% | 19.67% | 71.33% | 96.50% | 38.00% | - |
| `full_mix_s20_db20_gen80_zero2` | 0.04% | 19.50% | `think_unbalanced=54.33%` | 11.17% | 58.33% | 3.33% | 未完成 |

这些结果说明：

| 观察 | 含义 |
|------|------|
| LoRA D4 可以把 `finish_rate` 拉到 92%，且 BFCL 不掉 | 现有 DB 数据里有可学习的终止和工具策略信号 |
| full d2 ckpt150 也能学到 `set_knob_rate=98.50%` | full SFT 不是完全不能学习 DB agent 流程 |
| full 后续 checkpoint / full mix 出现复读、`finish_rate` 低、`<think>` 不闭合 | full 对样本格式和训练形态更敏感，容易把生成协议训坏 |
| full ckpt350 的 CMMLU、Multi-IF、BFCL 明显低于 base / LoRA | 问题不只是 DB 策略弱，还包含通用多轮指令和 tool/function 格式能力退化 |

因此 v3 不再继续优先验证“混入更多通用数据能不能救 full SFT”。v3 的核心问题改为：**当前 full SFT 是否因为整条多轮轨迹训练、历史 thinking 保留、终止监督权重过低，导致模型学成继续写轨迹/复读，而不是当前 turn 输出合法动作后停止。**

## 外部项目对照

本轮参考两个更贴近 Qwen3 agent full SFT 的开源项目：

| 项目 | 与本项目相关做法 | 对 v3 的启发 |
|------|------------------|--------------|
| EnvScaler | Qwen3 agent full SFT；使用 Qwen3 tool template；`mask_history=True`；把轨迹拆成当前 turn 监督；历史 assistant thinking 会移除 | full SFT 不直接训整条长轨迹；历史里不保留前面 assistant 的完整 thinking |
| CognitiveKernel-Pro | 发布 Qwen3-8B agent SFT；把轨迹拆成 `plan` / `action` / `end` 样本；输入只含 `recent_steps + progress_state`；显式 `stop(output, log)` | agent full SFT 更偏模块化/step-level 监督；终止是单独可学模块，不是靠整条长轨迹末尾自然学会 |

这两个项目不能直接证明我们的根因，但它们给出一个很强的工程先例：**成功的 Qwen3 agent full SFT 往往不会把完整多轮 agent 轨迹作为一条样本全部监督。**

## 实验目的

v3 只回答四个问题：

| 问题 | 判断依据 |
|------|----------|
| 整轨迹 full assistant loss 是否导致 continuation bias？ | step-level 后 `repetition`、`think_unbalanced`、`finish_rate` 是否显著恢复 |
| 历史 `<think>` 是否污染后续生成？ | remove-think-history 后 `<think>` 不闭合和复读是否下降 |
| 终止监督是否权重不足？ | end-step / finish oversample 后 `finish_rate` 是否上升，同时 DB avg 是否不明显下降 |
| full SFT 是否仍可作为后续路线？ | v3 最佳小规模 full 是否同时满足行为稳定和 DB eval 可接受 |

## 核心假设

| 假设 | 预期现象 | 验证方式 |
|------|----------|----------|
| `H1`：整条轨迹多 assistant loss 会诱导模型继续生成轨迹 | step-level current-turn 训练显著降低复读和未闭合 `<think>` | 对比 v2 full / v3 step-level 的 rollout 行为指标 |
| `H2`：历史 assistant thinking 是 full SFT 的主要污染源之一 | 去掉历史 `<think>` 后，`think_unbalanced` 和长篇复读下降 | 对比 `step_raw_history` 与 `step_remove_think_history` |
| `H3`：finish 信号在整轨迹中监督权重偏低 | 单独加入 end-step / oversample finish 后 `finish_rate` 上升 | 对比有无 `end_step` 增强 |
| `H4`：通用混合数据不是当前第一优先级 | 只改 DB 样本格式就能先恢复行为稳定性 | 先跑纯 DB v3，小规模通过后再考虑 general mix |

## 数据形态设计

### v2 当前形态

当前 full SFT 使用完整 `messages` 轨迹：

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "<think>...</think>\n<tool_call>...</tool_call>"},
    {"role": "tool", "content": "..."},
    {"role": "assistant", "content": "<think>...</think>\n<tool_call>...</tool_call>"},
    {"role": "tool", "content": "..."}
  ]
}
```

当前 TRL 训练会对完整轨迹里的所有 assistant tokens 做 assistant-only loss。风险是：模型同时学习多步操作、长 thinking、工具结果后的继续推理，以及最后很少出现的 `finish_tuning`。

### v3 目标形态

v3 将一条轨迹拆成多个 current-turn SFT 样本：

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "原始任务 + compact history / recent steps"},
    {"role": "assistant", "content": "<think>当前步分析</think>\n<tool_call>{...}</tool_call>"}
  ],
  "meta": {
    "source": "db_agent_v3",
    "step_idx": 3,
    "target_tool": "predict_performance",
    "is_final_step": false
  }
}
```

关键要求：

| 设计点 | v3 要求 |
|--------|---------|
| 监督粒度 | 只监督当前 step 的 assistant 输出 |
| 历史长度 | 默认保留最近 `k=4` 步，避免无限长上下文 |
| 历史 thinking | 默认移除历史 assistant `<think>...</think>`，只保留工具调用摘要和 tool response |
| 结束样本 | 单独保留 final step，并可 oversample |
| 格式 | 继续使用本项目 `<think>` + `<tool_call>` 协议，暂不切 Qwen3 原生 function call，以降低变量数 |
| 通用数据 | v3 第一阶段不混 general；先验证 DB 格式修复是否有效 |

## 实验矩阵

第一阶段只跑小规模，优先验证行为，不追求最终 DB avg。

这里把“展示顺序”和“执行顺序”分开：

| 排序口径 | 用途 |
|----------|------|
| 展示顺序 | 按当前最可能导致 full SFT 崩坏的原因排序，方便讨论根因 |
| 执行顺序 | 按训练成本和依赖关系排序，避免一上来跑高成本实验 |

### 展示顺序：按最可能原因排序

| 优先级 | 最可能原因 | 对应实验 | 数据规模 | 关键变量 | 预期能解释什么 |
|--------|------------|----------|---------:|----------|----------------|
| 1 | 整条轨迹 full assistant loss 诱导 continuation bias，模型学成“继续写下一轮轨迹” | `full_v3_a_step_raw_history_3k` | 3k 原轨迹拆 step | current-turn；历史 thinking 保留；无 finish 增强 | 如果行为明显恢复，说明主因是整轨迹训练形态 |
| 2 | 历史 assistant `<think>` 长链污染后续生成，放大复读和 `<think>` 不闭合 | `full_v3_b_step_no_think_history_3k` | 3k 原轨迹拆 step | current-turn；移除历史 thinking；无 finish 增强 | 如果明显优于 A，说明历史 thinking 是关键污染源 |
| 3 | `finish_tuning` / end-step 监督权重不足，模型会调参但不会停 | `full_v3_c_step_no_think_finish2x_3k` | 3k 原轨迹拆 step | 移除历史 thinking；final step 2x | 如果 `finish_rate` 大幅上升，说明终止监督不足成立 |
| 4 | 终止监督不足更严重，需要更强 end-step 重采样 | `full_v3_d_step_no_think_finish4x_3k` | 3k 原轨迹拆 step | 移除历史 thinking；final step 4x | 仅用于确认 2x 不够时是否还能救 finish |

### 执行顺序：按训练成本排序

| 顺序 | 实验 | 成本判断 | 继续条件 |
|------|------|----------|----------|
| 1 | `full_v3_a_step_raw_history_3k` | 最低成本；只做 step-level 拆分，不额外增加样本 | 如果相对 old full 无明显改善，说明仅 step-level 不够 |
| 2 | `full_v3_b_step_no_think_history_3k` | 仍是 3k 规模；只改变历史处理方式 | 如果 `think_unbalanced` 明显下降，说明历史 thinking 是关键因素 |
| 3 | `full_v3_c_step_no_think_finish2x_3k` | final step 重采样，训练 token 数上升但仍可控 | 如果 `finish_rate` 回升且 DB avg 不崩，作为 v3 主候选 |
| 4 | `full_v3_d_step_no_think_finish4x_3k` | 成本最高；只在 2x finish 仍不足时跑 | 仅当 2x finish 不足、但其他行为指标已明显改善时跑 |

第二阶段才扩到 6.6k 或重新考虑 general mix：

| 触发条件 | 下一步 |
|----------|--------|
| 3k v3 行为稳定但 DB avg 偏低 | 扩到 6.6k DB step-level |
| 3k v3 行为稳定但 CMMLU/BFCL 仍崩 | 在 v3 格式基础上再混 general |
| 3k v3 仍复读/不闭合 | 暂停 full SFT，回到 LoRA + RL 或改协议格式 |

## 数据构建规则

### Step 样本构建

对每条原始 DB 轨迹：

1. 保留 system prompt 和原始用户任务。
2. 找到每个 assistant turn。
3. 对第 `i` 个 assistant turn 构造一个样本：
   - 输入包含 system、原始 user、最近 `k` 个历史动作摘要、必要 tool response。
   - 输出只包含第 `i` 个 assistant turn。
4. 如果第 `i` 个 assistant turn 调用 `finish_tuning`，标记 `is_final_step=true`。
5. 过滤明显坏样本：
   - assistant 输出没有闭合 `<think>` / `<tool_call>`；
   - tool call JSON 不可解析；
   - 工具名不在 schema；
   - assistant 输出为空；
   - token length 超过 `8192`。

### 历史 thinking 处理

`remove_think_history=true` 时：

| 原历史内容 | v3 输入中保留 |
|------------|---------------|
| `<think>...</think>` | 删除 |
| `<tool_call>{...}</tool_call>` | 保留或转成简短 `Action: tool(args)` |
| tool response | 保留完整或截断到固定 token 上限 |
| final answer / finish | 保留简短终止标记 |

这样做的目的不是否定 thinking，而是避免模型在后续 step 的 prompt 里反复看到前面长 thinking，从而学习“继续写 thinking 历史”。

## 训练框架

v3 改用 LLaMA-Factory，不再使用 TRL。

原因不是“换一个训练壳”，而是为了尽量对齐 EnvScaler 的 Qwen3 agent full SFT recipe：

| 项 | v2 TRL full SFT | v3 LLaMA-Factory full SFT |
|----|-----------------|---------------------------|
| Qwen3 模板 | 使用本地 training chat template，依赖 TRL assistant mask | 使用 LLaMA-Factory `template: qwen3` |
| 历史 loss | 完整 messages 中所有 assistant turn 都可能被监督 | 使用 `mask_history: true`，只监督当前 assistant 输出 |
| agent 轨迹格式 | 一条完整多轮 DB 轨迹作为一个样本 | v3 预处理拆成 current-turn 样本 |
| thinking 历史 | 默认保留历史 `<think>` | v3 数据预处理移除历史 assistant thinking |
| 参考先例 | 与 EnvScaler 不完全一致 | 更接近 EnvScaler 的 full SFT 设置 |

因此 v3 的核心不是 LLaMA-Factory 本身，而是组合使用：

1. v3 step-level 数据；
2. `template: qwen3`；
3. `mask_history: true`；
4. `finetuning_type: full`；
5. 历史 assistant thinking 移除。

如果 LLaMA-Factory 本身需要的数据字段和当前 `messages` JSONL 不兼容，先写转换脚本，把 v3 step-level 数据转成 LLaMA-Factory `sharegpt` 格式，再训练。

### `full_v3_b_step_no_think_history_3k_zero3`

训练配置：

| checkpoint | `checkpoint-600` |

| max length | `8192` |
| lr | `0.000001` |
| eval 场景数 | `600` |

DB eval 结果：

| 指标 | 数值 |
|------|-----:|

| 平均步数 | `7.0` |
| predict 调用率 | `100.0%` |
| 结束原因 | `unknown=100.0%` |

轨迹行为稳定性诊断：

| checkpoint | repetition | trunc/protocol | bad_tool_rows | non_schema_rows | short abnormal | predict_rate | set_knob_rate | finish_rate | 结论 |
|------------|-----------:|---------------:|--------------:|----------------:|---------------:|-------------:|--------------:|------------:|------|
| `checkpoint-600` | `0.00%` | `14.50%` | `0.00%` | `0.00%` | `0` | `86.67%` | `100.00%` | `83.33%` | 复读、坏工具、非 schema 已消失，set knob 和 finish 明显恢复；但仍有 `<think>` 未闭合，predict 验证率低于目标 `90%` |

备注：

- 这是目前 DB eval 最强的 full SFT 结果，明显高于 LoRA D4 的 `8.81%` 和旧 full SFT 的 `5.45%`。
- `unknown=100.0%` 是 eval rollout JSONL 没有保存 `termination_reason` 的记录 bug，不应解读为模型真实结束原因。该问题已在 `d0a8639 Preserve eval rollout termination tracking` 修复；后续需要重跑 rollout 才能得到真实 `finish_tuning` / `max_turns_reached` 分布。
- `finish_rate=83.33%` 是从实际 `<tool_call>` 中反推得到，不依赖旧 report 的 `termination_reason` 字段。
- 该结果强烈支持 v3 主假设：full SFT 的主要问题至少部分来自整条多轮轨迹训练形态；step-level current-turn + remove-think-history 显著恢复 DB agent 任务收益和工具调用稳定性。
- 当前剩余问题集中在协议闭合和验证流程：`think_unbalanced=14.50%`、`predict_rate=86.67%`，后续需要重跑带 termination tracking 的 rollout，并考虑降低生成长度或强化 `<think>` 闭合 / predict-before-finish 约束。

## 判定逻辑

| 结果 | 结论 | 下一步 |
|------|------|--------|
| step-level 后复读和 `<think>` 不闭合大幅下降 | 整轨迹训练是主因之一 | 采用 v3 step-level 作为 full SFT 默认格式 |
| remove-think-history 后明显优于 raw history | 历史 thinking 污染成立 | 后续所有 full SFT 默认去历史 thinking |
| finish oversample 后 `finish_rate` 明显上升 | 终止监督权重不足成立 | v3 保留 final-step 增强 |
| 以上都无效 | 当前问题不主要来自样本组织 | 转查优化参数、模板、Qwen3 tool 格式或放弃 full SFT |
| 行为恢复但 benchmark 仍崩 | 需要在 v3 格式上再混通用 / function-calling replay | 设计 v3-general |

## 后续待办

### 国产卡复现实验与归因

当前 A800 + LLaMA-Factory + v3 step-level 数据已经显著恢复 full SFT 的 DB eval 和大部分轨迹行为，但还不能严格区分问题来源。后续计划在国产卡环境复现实验，用于判断退化来源到底是硬件/精度、训练框架、数据格式、模板/mask，还是这些因素叠加。

待验证问题：

| 问题 | 验证方式 | 判断依据 |
|------|----------|----------|
| 国产卡精度或算子是否导致 full SFT 更容易退化 | 在国产卡上复现 `full_v3_b_step_no_think_history_3k`，尽量保持数据、lr、global batch、max length 一致 | 如果同配置在国产卡显著复读/协议崩，而 A800 正常，优先排查硬件精度/算子 |
| 框架是否是主因 | 在同一硬件上对比 LLaMA-Factory v3 与 TRL v3 step-level | 如果 TRL v3 仍崩而 LLaMA-Factory v3 稳定，说明框架/template/mask 贡献较大 |
| 数据格式是否是主因 | 在同一框架上对比 v2 整轨迹与 v3 step-level | 如果 v2-style 仍复读而 v3 稳定，说明整轨迹监督粒度是主因 |
| 历史 thinking 是否是关键变量 | 对比 `v3_a_step_raw_history_3k` 与 `v3_b_step_no_think_history_3k` | 如果 raw history 的 `<think>` 不闭合/复读明显更高，说明历史 thinking 污染成立 |

注意：这组实验先作为归因验证，不直接追求更高 DB avg。优先记录 `repetition`、`think_unbalanced`、`bad_tool_rows`、`predict_rate`、`finish_rate` 和标准 benchmark，避免只看训练 loss 或 DB 平均收益。

## 当前结论

v3 的目标不是马上赢过 LoRA D4，而是先验证一个更基础的问题：**full SFT 是否必须使用更适合 agent 的 step-level / remove-think-history / end-step 监督格式。**

如果 v3 小规模能把 `finish_rate`、`think_unbalanced`、复读问题修回来，再谈扩规模和混通用数据；如果 v3 仍修不回来，继续在整轨迹 full SFT 上堆数据或调 checkpoint 意义不大。
