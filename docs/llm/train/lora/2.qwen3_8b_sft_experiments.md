#### 1. 先 merge LoRA

把下面命令里的 `<EXP>` 替换成：

- `a0_direct_only`
- `a1_full3k`
- `b1_depth_trimmed`
- `b2_depth_full`
- `c1_gain_natural`
- `c2_gain_balanced`

```bash
cd /root/private_data/DB-Opt-R1

BASE_MODEL=/root/private_data/Qwen3-8B
CKPT=/root/private_data/DB-Opt-R1/model_save/experiments/v2/sft/trl/lora/<EXP>/20260423_191024/checkpoint-270
MERGED_DIR=${CKPT}_merged

BASE_MODEL="$BASE_MODEL" CKPT="$CKPT" MERGED_DIR="$MERGED_DIR" python - <<'PY'
import os
import torch
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer

base = os.environ["BASE_MODEL"]
ckpt = os.environ["CKPT"]
out = os.environ["MERGED_DIR"]

tok = AutoTokenizer.from_pretrained(base, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
    base,
    dtype=torch.bfloat16,
    trust_remote_code=True,
    device_map="auto",
)
model = PeftModel.from_pretrained(model, ckpt)
model = model.merge_and_unload()

os.makedirs(out, exist_ok=True)
tok.save_pretrained(out)
model.save_pretrained(out)
print("merged saved to:", out)
PY
```

#### 3. 跑 eval

```bash
cd /root/private_data/DB-Opt-R1

RUN_NAME=$(date +%Y%m%d_%H%M%S)
OUT=/root/private_data/DB-Opt-R1/eval_results/v2/sft/<EXP>/${RUN_NAME}
mkdir -p "$OUT/rollouts" "$OUT/report"

python -m data_pipeline.synthesis.trajectory.sampler \
  --mode eval \
  --eval-questions /root/private_data/DB-Opt-R1/data_pipeline/data/eval/v2/eval_trajectories_v2.jsonl \
  --scenarios /root/private_data/DB-Opt-R1/data_pipeline/data/eval/v2/collected_eval_v2.json \
  --cost-model /root/private_data/DB-Opt-R1/cost_model/checkpoints/v10_lgbm \
  --knob-space /root/private_data/DB-Opt-R1/configs/knob_space.yaml \
  --output-dir "$OUT/rollouts" \
  --output-file sft_trajectories.jsonl \
  --model <SERVED_MODEL_NAME> \
  --api-base http://127.0.0.1:<PORT>/v1 \
  --api-key EMPTY \
  --parallel 64 \
  --api-max-concurrent 64 \
  --max-turns 10

python -m evaluate.run \
  --eval-data "$OUT/rollouts/sft_trajectories.jsonl" \
  --scenarios /root/private_data/DB-Opt-R1/data_pipeline/data/eval/v2/collected_eval_v2.json \
  --cost-model /root/private_data/DB-Opt-R1/cost_model/checkpoints/v10_lgbm \
  --knob-space /root/private_data/DB-Opt-R1/configs/knob_space.yaml \
  --output "$OUT/report" \
  --skip-bo
```

## 评估口径说明

2026-04-24 对 `eval report` 口径做了修正，并对本页结果全部重算：

- 主指标不再使用“最终累积 `set_knob` 后重跑 cost model”的口径
- 改为直接从 rollout 轨迹中的 `predict_performance` 提取：
  - `baseline_tps`
  - `best predicted_tps`
  - `best improvement_pct`
- `default_tps` 仍由 cost model 计算，但只按 `hardware + workload` 缓存

这意味着本页结果反映的是：

**每条 rollout 在搜索过程中找到的最佳配置表现**

而不是：

**模型最后停下来的那个配置表现**

## 核心结论

1. **SFT 相对 base 明显有效**
   最弱的 `c2_gain_balanced` 也有 `5.28%`，而 `qwen3_8b_base` 只有 `1.84%`。

2. **当前最优冷启动是 `b2_depth_full`**
   它在三条核心指标上都领先：
   - `avg vs 场景原始配置 = 7.88%`
   - `median vs 场景原始配置 = 0.68%`
   - `win rate vs 场景原始配置 = 62.0%`

3. **`B` 组说明：更深的成功轨迹有价值**
   - `b1_depth_trimmed = 6.59%`
   - `b2_depth_full = 7.88%`

4. **`A` 组说明：扩到 `full3k` 是有效的小改进**
   - `a0_direct_only = 5.56%`
   - `a1_full3k = 7.64%`

5. **`C` 组说明：自然收益分布优于强行均衡**
   - `c1_gain_natural = 6.76%`
   - `c2_gain_balanced = 5.28%`

6. **行为指标区分度仍然不强**
   除了 base 的平均步数更高（`9.5`），其余 SFT 实验大多稳定在 `7.9 ~ 8.2`。差异主要还是来自训练数据切法，而不是 rollout 轮数本身。

## 当前推荐

- **首选冷启动**：`b2_depth_full`
- **次优对照**：`a1_full3k`、`c1_gain_natural`
- **不建议继续投入**：`c2_gain_balanced`

如果后面继续做 RL / GRPO，最合理的起点就是：

- 主线：`b2_depth_full`
- 对照：`a1_full3k` 或 `c1_gain_natural`
