# Qwen3-8B LoRA SFT 数据规模实验设计

## 1. 实验目标

这组实验只回答一个问题：

- 在已经验证过的较优数据构成下，`Qwen3-8B + LoRA SFT` 到底需要多大规模的数据，收益才开始明显放缓。

因此这轮实验：

- **只改数据规模**
- **不再同时改数据分布**
- 继续沿用前面 `A / B` 组已经验证过的有效抽样思路
- 训练、评估、cost model、评估口径全部固定

规模档位沿用已有文档 [1.sft_rl_data_selection_design.md](/Users/caixiaomeng/Projects/Python/Compiler-R1/project/db-opt-r1/docs/llm/data/1.sft_rl_data_selection_design.md) 中“第三组：规模实验”的定义：

- `D1 = 1.5k`
- `D2 = 3k`
- `D3 = 4.5k`
- `D4 = 6k`

这里“规模”继续按样本条数定义，以便和旧设计保持一致；但每个规模档位都必须额外记录：

- 总 token 数
- assistant/tool 回合数统计

这样后面分析时可以确认：

- 本轮比较主要是在改样本规模
- 而不是因为某个规模意外吃进了明显更长的轨迹分布

## 2. 前置实验结论

这轮规模实验不是从零开始设计，而是建立在前面 6 组 `Qwen3-8B LoRA SFT` 消融结论之上。结果记录见 [2.qwen3_8b_sft_experiments.md](/Users/caixiaomeng/Projects/Python/Compiler-R1/project/db-opt-r1/docs/llm/train/lora/2.qwen3_8b_sft_experiments.md)。

只看“vs 场景原始配置”的平均提升，前面实验给出的结论是：

1. `A` 组结论：
   - `a1_full3k (7.64%) > a0_direct_only (5.56%)`
   - 含义不是“更多数据更好”，而是：
   - **在总样本数同为 3k 的前提下，加入少量 `retry_success` 比纯 `direct_success` 更好。**

2. `B` 组结论：
   - `b2_depth_full (7.88%) > b1_depth_trimmed (6.59%)`
   - 说明：
   - **保留更深的成功轨迹（tail）是有价值的。**

3. `C` 组结论：
   - `c1_gain_natural (6.76%) > c2_gain_balanced (5.28%)`
   - 说明：
   - **不需要强行做 gain bucket 配平，应保留自然收益分布。**

因此，规模实验的主 recipe 应该同时继承：

- `A` 组里“保留少量 retry-success 信号”
- `B` 组里“保留 depth-full tail 信号”
- `C` 组里“不要做人为 gain 配平”

## 3. 为什么统一 recipe 不需要再做 A×B 交叉

这一步必须先说清楚，不然后面的规模实验会写歪。

当前 `v2` manifest 标签逻辑定义在 [build_sft_manifests_v2.py](/Users/caixiaomeng/Projects/Python/Compiler-R1/project/db-opt-r1/data_pipeline/build_sft_manifests_v2.py)：

- `shape`：
  - `direct_success`
  - `retry_success`
- `depth_bucket`：
  - `main`
  - `tail`

对当前 `train-only v10` 池重新统计后，关键关系是：

- `direct_success + main = 6039`
- `direct_success + tail = 510`
- `retry_success + tail = 96`
- `tail total = 606`

也就是说：

- **所有 `retry_success` 都包含在 `tail` 里**
- 当前数据版本里不存在 `retry_success + main`

这意味着，把 `A` 组和 `B` 组结论合在一起时，不需要设计两条独立抽样线再交叉验证。因为：

- `A` 组希望保留的额外信号是 `retry_success`
- `B` 组希望保留的额外信号是 `tail`
- 而 `retry_success` 本身就是 `tail` 的一个子集

所以在当前数据版本下，**统一 recipe 直接选择“保留完整 tail 池”即可同时继承 `A` 和 `B` 两组的正向结论。**

## 4. 规模实验统一抽样 Recipe

### 4.1 Recipe 定义

本轮规模实验的统一 recipe 定义为：

1. 数据来源固定为：
   - `data_pipeline/data/train/v2/sft_trajectories_v10_train.jsonl`

2. 标签体系固定为：
   - `shape`
   - `depth_bucket`
   - `gain_bucket`
   - `workload`

3. 核心抽样规则：
   - **保留全部 `tail` 样本**
   - 其余 quota 从 `main` 样本中补齐
   - 从 `main` 补齐时，按 **`(workload, gain_bucket)` 分层抽样**
   - **不做 gain bucket 强行配平**

这套 recipe 的含义是：

- 用 `tail` 显式保留更深的成功轨迹
- 因为 `retry_success ⊂ tail`，所以也自然保留了 `A` 组验证过的 retry-success 信号
- 对剩余主干数据，保持自然收益分布而不是做人工均衡

### 4.2 与已有实验的关系

这套统一 recipe 在 `3k` 档位上，和现有的 `b2_depth_full` 是同一条主线：

- 都是“保留全部 tail，再从 main 中补齐”
- 都不做人为 gain 配平

因此：

- **`D2 = 3k` 可以直接视为当前 `b2_depth_full` 的规模锚点**
- 后续真正新增的规模实验，重点是：
  - `D1 = 1.5k`
  - `D3 = 4.5k`
  - `D4 = 6k`

## 5. 四个规模档位的构造方式

基于当前数据池：

- `main = 6039`
- `tail = 606`

四个规模档位的推荐构造如下：

### `D1 = 1.5k`

- 固定保留全部 `606` 条 `tail`
- 从 `main` 中分层抽样 `894` 条
- 总计 `1500` 条

### `D2 = 3k`

- 固定保留全部 `606` 条 `tail`
- 从 `main` 中分层抽样 `2394` 条
- 总计 `3000` 条

### `D3 = 4.5k`

- 固定保留全部 `606` 条 `tail`
- 从 `main` 中分层抽样 `3894` 条
- 总计 `4500` 条

### `D4 = 6k`

- 固定保留全部 `606` 条 `tail`
- 从 `main` 中分层抽样 `5394` 条
- 总计 `6000` 条

这 4 档在当前池上都是可行的，不需要为了 `6k` 再改 recipe。

## 8. 重点观察问题

这轮规模实验重点回答以下问题：

1. `1.5k -> 3k`
   - 数据量增加是否带来明显提升
   - 如果 `D1` 已经非常接近 `D2`，说明当前任务对 SFT 数据量不敏感

2. `3k -> 4.5k`
   - 现有 `3k` 是否已经接近平台
   - 还是继续增加样本还有可观收益

3. `4.5k -> 6k`
   - 更大规模是否继续带来收益
   - 如果收益非常有限，则后续不必继续扩大 LoRA SFT 冷启动数据集

最终要回答的不是“规模越大是不是越好”，而是：

- **在当前 recipe 下，`Qwen3-8B LoRA SFT` 的性价比拐点在哪。**

## 9. 预期结论形式

实验完成后，希望能得到下面三类结论：

### 9.1 是否存在明显规模收益

例如：

- `D1 < D2 < D3 ≈ D4`

这说明：

- 从 `1.5k` 增到 `4.5k` 有意义
- 但继续增到 `6k` 收益有限

### 9.2 `3k` 是否已经足够

如果 `D2` 与 `D3/D4` 差距很小，则说明：

- 现有 `3k` 已经是很强的 LoRA SFT 冷启动规模
- 后续更值得把算力放到 RL/GRPO，而不是继续堆 SFT 样本

### 9.3 是否值得为更大规模额外构数

如果 `D4` 显著优于 `D2`，那后面可以考虑：

- 在相同 recipe 上继续扩样本
- 或把更大规模版本作为 full SFT / RL 冷启动的上限数据集

## 10. 产物规划

建议这轮规模实验新增以下 manifest：

- `sft_manifest_d1_1p5k.jsonl`
- `sft_manifest_d2_3k.jsonl`
- `sft_manifest_d3_4p5k.jsonl`
- `sft_manifest_d4_6k.jsonl`

并同步输出：

- 每个 manifest 的 `rows`
- 每个 manifest 的总 token 数
- 每个 manifest 的 assistant/tool 回合统计
- `workload` 分布
- `gain_bucket` 分布
- `depth_bucket` 分布
- `shape` 分布

这里的目标不是再做一轮分布消融，而是确认：

- 随着规模变化，recipe 没有被实现细节悄悄改形

## 11. 执行顺序建议

为了减少重复训练，建议顺序如下：

1. 先确认 `D2` 是否直接复用 `b2_depth_full`
2. 新增构造 `D1 / D3 / D4`
3. 先跑 `D1`
4. 再跑 `D3`
5. 最后跑 `D4`
6. 与 `D2(b2)` 一起画规模曲线

这样做的好处是：

- 不重复跑已经有的 `3k` 锚点
- 最快看出“小规模是否已经够用”
- 如果 `D3` 已经开始明显平台，也可以决定是否还值得跑 `D4`

## 12. 本轮设计结论

这轮 `Qwen3-8B LoRA SFT` 规模实验的设计结论可以概括成一句话：

**在当前数据版本下，规模实验不再重新比较多条 recipe，而是在统一的 `A+B` 融合主线 recipe 上，仅研究 `1.5k / 3k / 4.5k / 6k` 的规模收益曲线。**

其中这条统一 recipe 的实际实现是：

- 保留全部 `tail`
- 自然保留其中的 `retry_success`
- 从 `main` 中按自然收益分布分层补齐
- 不做人为 gain 配平
