# SFT / RL 数据筛选设计

## 背景

当前项目已经具备：

- `raw env/task`：`collected_8c16g_hdd_20k.json`
- 多条 rollout 轨迹：`sft_trajectories_v10_part1.jsonl`、`sft_trajectories_v10_part2.jsonl`

这意味着数据筛选的核心问题已经不是“怎么造环境”，而是：

1. 如何从现有 rollout 中构造 **高质量 SFT 数据集**
2. 如何从现有 `env/task` 中抽取 **适合 RL 的训练集**
3. 如何保证 SFT 的 **多样性**，而不是只学到“一步调参即成功”的单一模式
4. 如何把“题难”与“协议没对齐”区分开，避免 RL 被无效失败污染

## 当前数据观察（基于 v10 两份轨迹）

### 数据规模

- 轨迹总数：`107,722`
- 覆盖场景数：`20,000`
- 平均每场景 rollout 数：`5.39`

### 成功/失败概况

- `improvement_pct <= 0` 的失败轨迹占比约 `69.25%`
- 全量场景按成功率分桶后大致为：
  - `easy`：约 `24.6%`
  - `frontier`：约 `46.8%`
  - `hard`：约 `14.9%`
  - `all_fail`：约 `13.7%`

### 失败原因概况

失败轨迹并不都是“任务难”，还包含大量协议类失败。按轨迹内容反推，失败大致分为：

- `protocol_unknown_tool`：约 `40.13%`
- `no_tool_call_end`：约 `31.38%`
- `clean_finish_no_gain`：约 `24.15%`
- `max_turn_like`：约 `3.37%`
- 其他：占比较小

这说明当前失败里，真正更像“题难”的只是其中一部分，另一大部分是 **动作空间未对齐**。

### 轨迹形态概况

正收益轨迹中：

- **一步就提升**：约 `96.5%`
- **多次尝试后才提升**：约 `3.5%`

这说明当前 SFT 数据天然偏向：

- 先看配置
- 调一次或少数几次
- 立刻 `predict`
- 提升后结束

而“失败后修正成功”的轨迹明显偏少。

### 重复样本概况

当前 raw rollout 里：

- `exact duplicate` 几乎没有
- 已确认出现的极少数完全重复样本，更像是**网络/请求失败后仍然落盘**的失败轨迹

因此：

- 原始层的完全重复，不是当前数据分布的主要问题
- 去重不应放在最前面全量做
- 更合理的顺序是：**先清洗失败轨迹和非法样本，再对候选训练集做去重**

### 正收益样本中的文本复读机概况

这里的“文本复读机”**只检查 assistant `content` 中单个 `<think>...</think>` 块内部的非结构化文本**。

明确不检查：

- `<tool_call>` JSON
- tool 返回
- 任何结构化字段
- 不跨两个不同的 assistant message 做重复匹配
- 不跨两个不同的 `<think>` 块做重复匹配

这里的“正收益”统一指：

- `best_predict_improvement > 0%`

在当前 **v2 母集正收益样本** 中：

- 正收益样本总数：`7,245`
- 其中包含 `<think>` 的样本：`7,245`
- 命中文本复读机的样本：`0`

按当前单个 `<think>` 块内部的更严格文本口径拆分：

- **字级连续重复**：`0`
- **词/短语级连续重复**：`0`
- **句子级重复**：`0`
- **段落级重复**：`0`

说明：

- 按当前更严格、也更符合本项目需求的定义，正收益样本里没有发现单个 `<think>` 块内部的文本复读机
- 前面观察到的少量重复，主要来自**跨 message / 跨 `<think>` 块**的重复，不属于当前这轮文本复读机统计口径
- 因此文本复读机在本轮不是主清洗矛盾

### 高收益离群点概况

当前数据里：

- `improvement_pct > 50%` 的样本共有 `4,778`
- 其中 `improvement_pct = 200%` 的样本共有 `4,428`

说明：

- `>50%` 这批不是普通“高收益长尾”，而是被 `200%` 截断值强烈主导
- 不能只按全局 `baseline_tps` 看是否异常，必须按 **`workload × 动作模式`** 的局部分布判断
- 对当前项目而言，`>50%` 应先视为 **待审计样本**，而不是直接进入主 `SFT / RL` 的标准正样本

### 当前 v2 重清洗结果

当前最终导出的主 SFT raw 候选集位于：

- `data_pipeline/data/pool/v2/sft_trajectories_v10_cleaned_pool.jsonl`
- `data_pipeline/data/pool/v2/sft_trajectories_v10_cleaned_stats.json`

基于该母集，进一步切出：

- `data_pipeline/data/eval/v2/*`：eval 集，共 `600` 条
- `data_pipeline/data/train/v2/sft_trajectories_v10_train.jsonl`：train-only 集，共 `6,645` 条

后续所有 **SFT 实验** 的数据构造，统一以 **train-only 的 `6,645` 条** 为基准，而不是以母集 `7,245` 条为基准。

使用的最终清洗逻辑是：

- 收益口径：轨迹内所有 `predict_performance` 返回中的 **best predict improvement**
- 起点：`best_predict_improvement > 0%`
- 剔除：assistant 中出现**不存在工具**的样本
- 剔除：协议错误 / tool call 坏格式样本
- 剔除：`restart` 未生效样本
- 剔除：没有 `finish_tuning` 的样本
- 剔除：没有 `<think>...</think>` 的样本
- 剔除：`high_gain_outlier`
- 最后收敛为：`1 env -> 1 best success trajectory`

当前 v2 统计结果：

- `best_predict_improvement > 0%`：`27,884`
- `drop_tool_hallucination`：`7,166`
- `drop_protocol_error`：`221`
- `drop_restart_invalid`：`3,016`
- `drop_no_finish_tuning`：`3,627`
- `drop_no_think`：`1`
- `drop_high_gain_outlier`：`2,396`
- `drop_same_env_lower_gain`：`4,212`
- 最终保留（母集）：`7,245`

## 设计目标

本设计希望达成以下目标：

1. `SFT` 不只学会“如何做对”，还要学会“如何在当前工具空间里稳定行动”
2. `SFT` 保留足够多的 **策略多样性**
3. `RL` 主训练集聚焦 **frontier 场景**
4. `RL` 尽量避免被协议类失败场景污染
5. `SFT` 与 `RL` 在 task 级别允许部分重叠，但在轨迹级别不复用同一条样本

## 非目标

- 本设计不讨论在线 rollout infra
- 不讨论 reward 函数实现细节
- 不讨论环境生成
- 不要求一次性消除所有协议类失败，但要求区分“中途协议出错后恢复成功”和“一路协议失败到结束”

## 基本定义

### 1. 轨迹级标签

对每条 rollout 轨迹打以下标签：

#### `has_protocol_error`

任意命中：

- 出现 `Unknown tool`
- 出现 `Invalid tool call format`
- 或出现终止前明显协议错误

说明：这是一个**出现过协议错误**的标签，不是直接过滤条件。

#### `protocol_failure_recovered`

满足：

- `has_protocol_error = true`
- 协议错误之后，轨迹重新回到当前合法工具空间
- 最终仍然形成有效后续步骤，并达到 `success_strong / success_weak / clean_no_gain` 之一

说明：这类轨迹不能因为“出现过协议错误”就直接删掉。

#### `protocol_failure_unrecovered`

满足：

- `has_protocol_error = true`
- 协议错误之后没有恢复到合法工具使用流程
- 或最终以协议错误/无效终止收尾

说明：这类轨迹才应从主 SFT / 主 RL 分析中剔除。

#### `success_strong`

- `best_predict_improvement > 3%`

说明：与当前采样命令中的 `--good-threshold 0.03` 对齐。

#### `success_weak`

- `0 < best_predict_improvement <= 3%`

说明：表示有正向信号，但未达到当前 SFT 正样本阈值。

#### `direct_success`

- 第一次 `predict_performance` 就出现正收益，且轨迹的 best predict 来自第一次 `predict_performance`

#### `retry_success`

- 前面至少一次 `predict_performance <= 0`
- 后面某次才第一次出现正收益，且轨迹的 best predict 不来自第一次 `predict_performance`

#### `clean_no_gain`

- 没有 `protocol_failure_unrecovered`
- 轨迹中走到了 `predict_performance`
- 最终 `best_predict_improvement <= 0`

### 2. 场景级指标

对每个 `env_sample_idx` 聚合：

- `rollout_count`
- `clean_rollout_count`
- `protocol_error_rate`
- `unrecovered_protocol_rate`
- `strong_success_rate = #success_strong / clean_rollout_count`
- `weak_success_rate = #(success_strong or success_weak) / clean_rollout_count`
- `retry_success_count`
- `best_improvement`
- `avg_valid_tool_calls`

## 难度定义

本项目中，**任务难度的主定义是场景级成功率，不是单条轨迹长度。**

### 场景难度桶

基于 `strong_success_rate`：

- `easy`
  - `strong_success_rate > 0.8`
- `frontier`
  - `0.1 <= strong_success_rate <= 0.8`
- `hard_but_learnable`
  - `strong_success_rate = 0`
  - 但 `weak_success_rate > 0`，或 `best_improvement > 0`
- `all_fail`
  - `weak_success_rate = 0`
- `protocol_heavy`
  - `protocol_error_rate > 0.25`

说明：

- `frontier` 是 RL 主力池
- `hard_but_learnable` 可少量混入 RL
- `all_fail` 和 `protocol_heavy` 不进入 RL 主池

注：

- 上面是**通用场景桶定义**
- 当前已经落地的 **`RL v2` 实际采样口径**，以后文 `RL 数据集构造` 章节为准

## 多样性原则与实验变量

本项目里，**多样性很重要，但不能无边界扩张**。  
应先区分两类维度：

1. **基础覆盖项**：必须被控制住，但不作为主要实验变量
2. **核心消融项**：真正值得花算力做对照实验的变量

### 基础覆盖项（固定约束，不作为主实验变量）

这些维度应在所有 SFT 数据集中尽量保持基本覆盖，否则实验结论会被混淆。

#### 1. workload 覆盖

- `read_only`
- `write_heavy`
- `mixed`
- `high_concurrency`

要求：每个实验集都不能严重偏科到单一 workload。

#### 2. 动作模式覆盖

按 `set_knob` 涉及的参数族做粗分类：

- memory 类
- wal/checkpoint 类
- io/cost 类
- 并发类
- autovacuum 类

要求：这些类别要有基本覆盖，但**不建议把动作模式本身作为第一轮主消融变量**。

#### 3. 重启正确性

`restart_pg` 不是多样性维度，而是**环境一致性约束**。

要求：

- 对涉及 `restart: true` knob 的轨迹，若在 `predict_performance` 前缺少 `restart_pg`，应视为 **bug 样本**，从 SFT 主实验集排除
- 这一类不参与“多样性”讨论

### 核心消融项（建议真正做实验）

结合当前项目目标，主实验只保留 **3 组主轴**：

1. **SFT 样本多样性**
2. **SFT 难度分布**
3. **SFT 样本规模**

重复性当前只保留现状说明，不再作为单独实验轴。

其中，“多样性”不是无限扩张，而是只围绕真正值得花算力验证的维度展开。

第一轮多样性实验只围绕以下 3 条内部子轴展开：

#### 1. 轨迹形态

- `direct_success only`
- `当前 train-only 的完整 6645 条`

核心问题：

- 只保留 `direct_success` 是否会让策略更单一
- 把当前 train-only 的完整 `6,645` 条一起放进来，是否能带来额外收益

#### 2. 轨迹深度

建议用以下代理信号：

- `predict_performance` 调用次数
- 合法工具调用数
- `set_knob` 次数

核心问题：

- 去掉深度长尾后，性能是否更稳
- 保留深度长尾，是否能带来额外泛化

当前深度长尾定义建议固定为：

- `predict_count >= 2`
  或
- `valid_tool_call_count >= 9`
  或
- `set_knob_count >= 2`

按当前 **train-only** 数据统计：

- 深度长尾样本：`606`
- 去掉深度长尾后的主体样本：`6,039`

#### 3. 收益分布

不再使用旧的 `3~10 / 10~30 / 30%+` 分桶。

当前 `v2` 的实际收益分布应按下面四档切：

- `0% ~ 1%`
- `1% ~ 3%`
- `3% ~ 10%`
- `10% ~ 50%`

核心问题：

- 当前自然分布是否已经足够好
- 对主收益区间重新配平后，是否能提升泛化

按当前 **train-only** 数据统计：

- `0% ~ 1%`：`1,442`
- `1% ~ 3%`：`1,589`
- `3% ~ 10%`：`2,098`
- `10% ~ 50%`：`1,516`

## SFT 数据集构造

### 目标

构造一个 **高质量、协议稳定、策略多样** 的 SFT 集，不只教“答案”，也教“正确行动方式”。

### 推荐组成

当前这一轮实验，SFT 只使用清洗后的 `SFT-Quality` 主池，不引入 `clean_no_gain` 或其他对齐池。

#### A. `SFT-Quality`

主 SFT 数据，要求：

- 不属于 `protocol_failure_unrecovered`
- `has_protocol_error = false`
- `has_tool_hallucination = false`
- `has_think = true`
- `best_predict_improvement > 0`
- `has_finish_tuning = true`
- 不属于 `high_gain_outlier`

默认每个场景只保留 `1` 条主轨迹：

- 优先保留 `improvement_pct` 最高的成功轨迹

说明（项目假设，不是论文定论）：

- 对主 SFT 而言，`1 env -> 1 best success trajectory` 即可
- 若同一场景存在多个 success，但最终收益差距明显（例如 `5%` vs `50%`），则**只保留最高收益轨迹**
- 较弱成功轨迹若要利用，只能进入单独的对照实验、偏好数据或分析集，不能与主成功轨迹等权混入 SFT

当前不单独构造 `SFT-Align`，也不研究 `clean_no_gain` 是否有帮助。

### SFT 抽样规则

建议采用 **分层配额抽样**：

先按：

- workload（基础覆盖项）
- 动作模式（基础覆盖项）
- 轨迹形态（核心变量）
- 收益带宽（核心变量）
- 探索深度（核心变量）

打标签，再按配额采样，而不是简单按 reward 排序截断。

同时增加三条硬约束：

- 涉及 `restart: true` knob 且 `predict_performance` 前无 `restart_pg` 的样本，不进入主 SFT 集
- 没有 `finish_tuning` 的样本，不进入主 SFT 集
- 没有 `<think>...</think>` 的样本，不进入主 SFT 集
- 收益统一按轨迹内 `predict_performance` 的 **best predict improvement** 计算

### SFT 不应直接纳入的样本

- `protocol_failure_unrecovered`
- 明显混入其他 agent schema 的轨迹
- `no_think`
- `no_finish_tuning`
- `high_gain_outlier`

## RL 数据集构造

### 目标

从现有 `env/task` 中离线抽取 **适合 RL 的 task set**，而不是把所有场景一股脑塞进去。

### 基本原则

- RL 主池只看 **场景级 best predict 分桶**
- RL 不要求 `SFT` 那样的整轨质量
- RL 数据集来源于 **原始 `v10 part1/part2` 轨迹池**，不是当前 `train-only` 的 `6645` 条
- RL 只保留 `1q/env`
- RL 只额外做两类清洗：
  - 排除 `eval/v2` 已占用的 `env`
  - 排除 `high_gain_outlier` 的 `env`

### RL 数据来源

当前 `v2` RL 数据集直接来源于：

- `data_pipeline/data/train/sft_trajectories_v10_part1.jsonl`
- `data_pipeline/data/train/sft_trajectories_v10_part2.jsonl`

而不是来源于：

- `data_pipeline/data/train/v2/sft_trajectories_v10_train.jsonl`

原因：

- `train-only` 已经被压成 `1 env -> 1 best success trajectory`
- 这更适合 `SFT`
- `RL` 训练真正需要的是：
  - `system prompt`
  - `question`
  - `env_sample_idx`
  - 以及该 `env` 对应的 reward 反馈

因此，`RL` 应从原始轨迹池里重建 prompt/task pool，而不是复用 `SFT` clean train。

### RL v2 的实际清洗规则

`RL` 当前采用的清洗口径是：

1. 先排除 `eval/v2` 已占用的 `600` 个 `env`
2. 只保留存在合法：
   - `system`
   - `question`
   - `env_sample_idx`
   的轨迹记录
3. 不再按 `SFT` 标准清：
   - `no_think`
   - `no_finish_tuning`
   - 协议/幻觉工具等整轨质量问题
4. 但仍然做 **`high_gain_outlier` 的 env 级剔除**
   - 只要某个 `env` 出现过 `high_gain_outlier` 轨迹
   - 该 `env` 就不进入主 `RL` 池
5. 最后对每个保留 `env` 只采：
   - `1q/env`

说明：

- `RL` 的场景难度只由该 `env` 在原始轨迹池中的 **best predict improvement** 分布决定
- 中间轨迹是否有协议错误、是否有 `<think>`、是否显式 `finish_tuning`
  在当前 `RL v2` 里都不作为主过滤条件
- 唯一必须额外隔离的是：
  - `eval` 泄露
  - `high_gain_outlier`

### RL 场景分桶口径

按每个 `env` 在原始轨迹池中的 best predict 统计：

- `strong_success_rate = #(best_predict_improvement >= 3%) / rollout_count`
- `weak_success_rate = #(best_predict_improvement > 0%) / rollout_count`

然后分桶：

- `easy`
  - `strong_success_rate > 0.8`
- `frontier`
  - `0.1 <= strong_success_rate <= 0.8`
- `hard_but_learnable`
  - `strong_success_rate = 0`
  - 但 `weak_success_rate > 0`
- `all_fail`
  - `weak_success_rate = 0`

### RL 主训练集：`RL-Frontier`

入池条件建议：

- `0.1 <= strong_success_rate <= 0.8`
- `has_high_gain_outlier = false`
- 不属于 `eval/v2`
- `1q/env`

说明：

- 这批场景“有时能做对，有时做不对”
- 最适合 RL 提高稳定性

### RL 辅助训练集：`RL-Hard`

入池条件建议：

- `strong_success_rate = 0`
- `weak_success_rate > 0`
- `has_high_gain_outlier = false`
- 不属于 `eval/v2`
- `1q/env`

比例建议：

- 占 RL 总 task pool 的 `10% ~ 20%`

作用：

- 防止 RL 只学会简单 frontier
- 让模型接触一部分更难但仍可学的场景

### RL 不进入主池的场景

#### `easy`

- `strong_success_rate > 0.8`

问题：

- reward 饱和
- 浪费 rollout 预算

#### `all_fail`

- `weak_success_rate = 0`

问题：

- 当前阶段几乎无学习信号

#### `protocol_heavy`

问题：

- 在当前 `RL v2` 里不再单独按协议比例切桶
- 协议问题不作为主过滤条件
- 因为当前 RL 只关心：
  - `question`
  - `env`
  - reward 能否提供学习信号

## SFT 与 RL 要不要交叉

### 结论

**要交叉，但只做 task 级别部分交叉，不做轨迹级复用。**

### 推荐策略

- 同一个 `env/task` 可以同时：
  - 贡献高质量成功轨迹给 `SFT`
  - 作为 `frontier` 任务进入 `RL`

但：

- 不把同一条轨迹同时作为：
  - SFT 训练样本
  - RL 难度评估依据

### 原因

如果完全切开：

- SFT 学到的 task 分布
- RL 看到的 task 分布

会断层太大。

如果完全重叠且全量进入 RL：

- easy 场景浪费预算
- protocol-heavy 场景污染训练

因此最佳方案是：

- **同源 task pool**
- **部分重叠**
- **按规则抽取 RL 子集**

## SFT 实验设计

本项目的主实验矩阵只分成 3 组：

1. **多样性实验**
2. **难度实验（待单独讨论）**
3. **规模实验**

重复性不再单独设计实验。

整体原则是：

- **先固定基础覆盖项**
- 每次只改一组主轴
- 其余主轴保持不变

### 实验总原则

所有 SFT 实验都应满足：

1. train / val / test 按 `env_sample_idx` 切分，避免同场景泄漏
2. workload 覆盖基本一致
3. 动作模式覆盖基本一致
4. 重启 bug 样本全部排除
5. 总 token 预算尽量一致
6. 训练超参数保持一致

### 第一组：多样性实验

这组实验回答的是：

- 只保留 `direct_success` 是否就够了
- 深度长尾样本要不要保留
- 当前自然收益分布要不要重新配平

这里的“多样性”只讨论：

- 轨迹形态
- 轨迹深度
- 收益分布

不讨论：

- workload 作为实验变量
- 动作模式作为实验变量
- 重启正确性作为实验变量

这些都属于基础覆盖或 bug 清洗约束。

#### 多样性实验总原则

- 难度分布固定为同一版基准配置
- 所有 `A / B / C` 组实验统一固定为 **`3k` 样本**
- 优先控制 **总 token budget** 一致；若工程上暂时做不到，则至少固定 **optimizer steps / global batch / max length** 一致
- `A / B / C` 组不允许同时改变“数据特征”和“规模”两个变量
- 每个场景最多 `1` 条主轨迹
- 当前所有 SFT 实验统一基于 **train-only `6,645` 条**，不使用母集 `7,245` 条

#### 第一阶段：轨迹形态消融

#### `SFT-A0-DirectOnly`

- 只保留 `direct_success`
- 实验集大小固定为：`3,000`

作用：

- 测试“只教一步命中”是否已经足够
- 由于它与全量 train-only 只相差 `96` 条，因此这更适合作为**小型 side ablation**，不宜当作主实验轴投入过多算力

#### `SFT-A1-Full3k`

- 使用当前 **train-only pool** 的 `3,000` 条子集
- 包含：
  - `direct_success`
  - 少量 `retry_success`
  - 少量其他正收益轨迹

作用：

- 测试把非 `direct_success` 的少量轨迹一并纳入后，是否有额外收益

实施约束：

- `A0` 与 `A1` 都固定为 **`3,000` 条**
- 推荐做法：
  - `A0`：从 `direct_success` 中分层抽样 `3,000` 条
  - `A1`：保留全部 `96` 条 `retry_success`，再从 `direct_success` 中抽样 `2,904` 条，组成 `3,000` 条
- 由于 `retry_success` 数量很少，这组实验只回答：
  - “少量 retry 样本是否值得保留”
  - 不回答“重试形态是否是主导因素”

#### 第二阶段：轨迹深度消融

#### `SFT-B1-DepthTrimmed`

- 去掉“深度长尾”样本
- 实验集大小固定为：`3,000`

作用：

- 测试去掉多步探索尾部后，模型是否更稳

#### `SFT-B2-DepthFull`

- 保留深度长尾样本，并与主干样本混合成 `3,000` 条
- 当前深度长尾池大小：`606`

作用：

- 测试深度长尾是否真的带来额外收益

实施约束：

- `B1` 与 `B2` 都固定为 **`3,000` 条**
- 推荐做法：
  - `B1`：从非长尾样本中分层抽样 `3,000` 条
  - `B2`：保留全部 `606` 条长尾样本，再从非长尾样本中抽样 `2,394` 条，组成 `3,000` 条
- 这样比较的是：
  - “是否引入深度长尾”
  - 而不是“是否喂了更多数据”

#### 第三阶段：收益分布消融

#### `SFT-C1-GainNatural`

- 使用当前自然收益分布
- 实验集大小固定为：`3,000`

#### `SFT-C2-GainBalanced`

- 按当前四个真实收益桶做重采样：
  - `0% ~ 1%`
  - `1% ~ 3%`
  - `3% ~ 10%`
  - `10% ~ 50%`
- 在 train-only 上，这四个桶分别为：
  - `1,442`
  - `1,589`
  - `2,098`
  - `1,516`
- 当前 `3k` 配平方案可直接取：
  - 每桶 `750`
  - 共 `3,000` 条

作用：

- 测试自然分布与收益配平分布谁更适合当前任务

实施约束：

- `C1` 与 `C2` 都固定为 **`3,000` 条**
- 推荐做法：
  - `C2`：每个收益桶各取 `750` 条，共 `3,000` 条
  - `C1`：从自然分布中抽取 `3,000` 条
- 这样比较的是：
  - “自然收益分布 vs 配平收益分布”
  - 而不是“更多或更少样本”

### 第二组：难度实验

这组先不在本轮文档里定具体配方，后面单独讨论。

### 第三组：规模实验

这组实验回答的是：

- 在最佳多样性配方下，SFT 到底需要多大规模
- 更大规模带来的收益，是不是值得额外训练成本

#### 规模实验总原则

- 使用前两组里最优的 recipe
- 只改样本规模，不改数据分布

在前面最佳多样性配方上做规模实验：

- `SFT-D1-1p5k`
- `SFT-D2-3k`
- `SFT-D3-4p5k`
- `SFT-D4-6k`

推荐规模设置：

- `D1`：`1.5k`
- `D2`：`3k`
- `D3`：`4.5k`
- `D4`：`6k`

说明：

- `D` 组只研究规模，因此保持同一最佳 recipe 的采样规则不变
- 若最佳 recipe 在严格配平下无法刚好满足 `6k`，则允许做**最接近 `6k` 的近似构造**，但需在实验记录中注明

建议不是直接按“条数”，而是按：

- 总 token
或
- 总 assistant/tool 回合数

来控规模。

## 推荐的实验顺序

不建议一开始就做全矩阵。推荐顺序：

1. **多样性实验**
   - 先做 `B1 / B2`
   - 再做 `C1 / C2`
   - `A0 / A1` 作为小型 side ablation 放在后面
2. **难度实验**
   - 后面单独讨论后再定
3. **规模实验**
   - 在最佳多样性配方上做 `D1 / D2 / D3 / D4`

这样可以避免：

- 一次展开太多变量
- 算力被组合爆炸吃掉
- 最后无法解释哪条 trick 真有效

## 实验前置：数据清洗

在做任何 `SFT` / `RL` 抽样和消融实验之前，必须先把原始 rollout 清洗成**可分析、可比较、可复现**的中间数据层。

否则会出现三个问题：

1. 脏样本和正常失败混在一起，导致“难度”统计失真
2. 不同实验集的 bug 样本比例不同，导致实验结论不可信
3. 候选训练集还没收敛时就全量去重，会浪费资源且意义不大

### 清洗目标

把原始 rollout 分成三层：

1. **原始层**
   - 保留完整 JSONL，不做覆盖
2. **清洗层**
   - 统一格式、补标签、隔离 bug/协议错误
3. **实验层**
   - 基于清洗层再切 `SFT/RL/manifest`

### 清洗步骤

#### Step 1. 轨迹规范化

对每条轨迹做统一解析，提取标准字段：

- `sample_id`
- `env_sample_idx`
- `messages`
- `tool_sequence`
- `set_knob_signature`
- `valid_tool_call_count`
- `predict_count`
- `finish_tuning`
- `has_think`
- `improvement_pct`
- `raw_reward`

目标：

- 为后续标签、候选集构造和最终去重提供统一视图

#### Step 2. 协议清洗

对每条轨迹打协议类标记：

- `has_unknown_tool`
- `has_invalid_tool_format`
- `has_no_tool_call_end`
- `has_cross_schema_tool`

然后产出：

- `has_protocol_error`
- `protocol_failure_recovered`
- `protocol_failure_unrecovered`

处理原则：

- 协议失败不等于任务难
- 不能因为“出现过协议错误”就直接删掉样本
- 只有 `protocol_failure_unrecovered` 才从主 SFT/RL 池隔离
- 对主 `SFT` 再增加一条硬约束：
  - **只要 assistant 中出现不存在的工具调用（tool hallucination），该轨迹就不进入 SFT**
  - 原因不是“它最终有没有恢复”，而是**不存在的工具不能被模型学进去**

#### Step 2.5. 输出格式一致性清洗

对主 `SFT` 再增加一个格式约束：

- `has_think`

处理原则：

- 主 `SFT` 只保留 assistant 明确包含 `<think>...</think>` 的轨迹
- 即使 assistant 里有自然语言说明，但如果没有 `<think>` 标签，也视为**格式不一致样本**
- `no_think` 不进入主 `SFT`

说明：

- 这里讨论的是**训练格式一致性**，不是内容质量高低
- `no_think` 样本可能内容正常，但它会破坏当前训练模板的一致性

#### Step 3. 参数生效一致性清洗

这里不再单独讨论“重启模式多样性”，而是直接做一个更简单的合法性判定：

**每条样本里，凡是被设置过、并会影响后续 `predict_performance` / `finish_tuning` 判断的参数，都必须在当前环境语义下已经生效。**

实现上只需要做一件事：

- 顺着轨迹扫描 `set_knob / restart_pg / reset_config / predict_performance / finish_tuning`
- 检查每次 `predict_performance`（以及必要时 `finish_tuning`）发生时，之前设置过的参数里是否仍有 **未生效参数**

最典型的非法情况就是：

- 改了 `restart: true` 的 knob
- 但在 `restart_pg` 前就去 `predict_performance`

此时该样本直接判为：

- `config_effective = false`

否则：

- `config_effective = true`

处理原则：

- `config_effective = false` 的轨迹视为 **bug 样本**
- 不进入主 SFT 集
- 不进入主 RL 集
- 默认从主实验数据层剔除

说明：

- 这一步本质上是在判断：**样本里 agent 看到的“当前配置”是否合法**
- 不需要额外把它设计成一整套复杂的多维标签
- 这一步属于环境一致性清洗，不属于“多样性”或“难度”分析

#### Step 4. 合法失败与真实难样本区分

对 `improvement_pct <= 0` 的轨迹继续分层：

- `clean_no_gain`
- `max_turn_like`
- `protocol_failure_unrecovered`

目标：

- 只把 `clean_no_gain` 这类真实失败留给难度分析
- 不让**未恢复**协议失败污染 hard/frontier 统计

#### Step 5. 高收益离群点清洗

这里的目标不是否定所有高收益样本，而是把**更像异常恢复、比例失真或 cost model 过冲**的样本从主 `SFT / RL` 训练池中剥离出来。

先定义：

- `high_gain_candidate`
  - `improvement_pct > 50%`

对每条 `high_gain_candidate`，再做三步判断。

##### 5.1 低基线过滤

- 以当前 cost model、官方默认配置、场景自身 `workload + hardware` 为输入
- 计算：
  - `default_baseline_tps`

若：

- `baseline_tps < default_baseline_tps`

则标记：

- `baseline_below_default = true`

说明：

- 这类样本更像“起点本身已经退化/异常”，不属于我们要学习的常规性能调优轨迹

##### 5.2 预测过冲过滤

计算：

- `pred_over_actual = predicted_tps / max(actual_tps, 1)`
- `pred_over_baseline = predicted_tps / max(baseline_tps, 1)`

然后按 **`workload × 动作模式`** 取参考上界。

参考集统一使用：

- `3% <= improvement_pct <= 50%`
- `config_effective = true`
- 非 `protocol_failure_unrecovered`
- `actual_tps > 1`
- `baseline_tps > 1`

若某个 `workload × 动作模式` 桶参考样本数足够（建议 `>= 30`），则使用该桶的：

- `pred_over_actual_p99`
- `pred_over_baseline_p99`

若该桶样本太少，则退回到 **`workload` 级** 的：

- `pred_over_actual_p99`
- `pred_over_baseline_p99`

若：

- `pred_over_actual > 上界`
  或
- `pred_over_baseline > 上界`

则标记：

- `prediction_overshoot = true`

##### 5.3 封顶高收益过滤

若：

- `improvement_pct = 200%`

则直接标记：

- `capped_high_gain = true`

说明：

- `200%` 是系统截断上限，不是真实提升值
- 这类样本已经信息丢失，不适合作为主训练样本

##### 5.4 最终判定

只要满足以下任一条件，即标记：

- `high_gain_outlier = true`

条件：

- `capped_high_gain = true`
- `baseline_below_default = true`
- `prediction_overshoot = true`

处理原则：

- `high_gain_outlier = true` 的轨迹：
  - 不进入主 `SFT`
  - 不进入主 `RL`
  - 进入 `audit_high_gain` / `holdout`

当前 v10 两份轨迹按这套规则统计：

- `best_predict_improvement > 0%` 的样本共有 `27,884`
- 其中：
  - `tool_hallucination` 剔除 `7,166`
  - `protocol_error` 剔除 `221`
  - `restart_invalid` 剔除 `3,016`
  - `no_finish_tuning` 剔除 `3,627`
  - `no_think` 剔除 `1`
  - `high_gain_outlier` 剔除 `2,396`
  - `same_env_lower_gain` 剔除 `4,212`
- 清洗后剩余可用正收益样本：
  - `7,245`

#### Step 6. 场景级可用性清洗

聚合到 `env_sample_idx` 后，先筛掉明显不适合作为训练基座的场景：

- `clean_rollout_count = 0`
- `protocol_error_rate` 过高
- 全部 rollout 都属于参数未生效 bug 样本
- 存在 `high_gain_outlier` 轨迹
- 场景元数据缺失严重

目标：

- 保证后续 `env_labels` 是建立在可用 rollout 之上的

#### Step 7. 固定 train / val / test 划分

在清洗后、做实验前，按 `env_sample_idx` 固定划分：

- `train_env_ids`
- `val_env_ids`
- `test_env_ids`

要求：

- 同一场景的不同 rollout 不得跨 split
- 后续所有实验共用同一套 split

#### Step 8. 候选训练集构造

先基于清洗后的轨迹/场景标签，构造：

- `SFT-Quality` 候选池
- `RL-Frontier` 候选场景池
- `RL-Hard` 候选场景池

说明：

- 这一步之后，样本规模已经大幅收缩
- 许多“网络失败后落盘”的空样本、协议未恢复样本、配置未生效 bug 样本会先被排掉
- `high_gain_outlier` 也会在这一步之前被隔离到 `audit_high_gain`

### 清洗后推荐产物

建议在实验前先固定产出：

- `trajectory_clean.parquet`
- `trajectory_config_invalid.parquet`
- `trajectory_protocol_failure.parquet`
- `trajectory_high_gain_audit.parquet`
- `env_clean.parquet`
- `train_env_ids.json`
- `val_env_ids.json`
- `test_env_ids.json`

其中：

- `trajectory_clean` 是所有后续 `SFT` manifest 的基座
- `env_clean` 是所有后续 `RL` 和难度分析的基座

### 清洗后的数据流

推荐流程：

```text
raw jsonl
-> trajectory normalization
-> protocol / restart / failure / high-gain labeling
-> env aggregation
-> clean train/val/test split
-> candidate SFT / RL pool construction
-> SFT ablations / RL ablations
```

### 设计要求

后续所有实验设计都默认建立在以下前提上：

- 已完成协议清洗
- 已完成参数生效一致性清洗
- 已完成高收益离群点清洗
- 已固定场景级 split

若没有这一步，后面的：

- 多样性实验
- 难度实验
- 规模实验

都不应直接开始。

- workload 分布
- 轨迹形态分布
- knob 类别覆盖
- `retry_success` 占比
- 平均合法工具调用数

### RL 侧

- `frontier / hard / easy / all_fail` 占比
- `dropped_high_gain_outlier_envs`
- 平均 `strong_success_rate`
- 不同 workload 的覆盖是否均衡

## 推荐落地顺序

1. 先完成数据清洗，产出 `trajectory_clean` 和 `env_clean`
2. 给清洗后的轨迹打轨迹级标签
3. 再聚合出 `env_labels`
4. 固定 train / val / test 场景划分
5. 先产候选三桶：
   - `SFT`
   - `RL`
   - `Holdout`
6. 只对候选训练集做最终去重 / 相似度控制
7. 验证分布是否明显优于当前全量混用
8. 再细化成：
   - `SFT-Quality`
   - `RL-Frontier`
   - `RL-Hard`

## 最终建议

当前项目最不该做的，是：

- 用 `reward > 0` 一把梭切 SFT
- 用剩下的全塞 RL

更合理的方式是：

- **SFT 学高质量成功轨迹，并有意识补充失败后修正成功的多样性**
- **RL 从原始 `v10` 轨迹池重建 prompt/task pool，而不是复用 SFT clean train**
- **RL 主池训练同源场景中的 `frontier` 子集，并排除 `high_gain_outlier` env**
- **RL 辅助池可少量混入 `hard_but_learnable`**
